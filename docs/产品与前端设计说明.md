# 陪玩服务信息平台 · 产品与前端设计说明

> 本文给参与界面/交互设计的同学使用。聚焦「这是什么产品、有哪些角色、每个角色要完成什么任务、需要哪些页面」，不涉及代码实现细节。
> 当前前端只搭了基础框架（约 1/3），核心业务页面尚未设计，正是从设计侧重新定义体验的好时机。

---

## 1. 产品是什么

**陪玩服务信息平台**：连接「玩家」「陪玩服务者」「陪玩工作室」三方的信息管理平台。

它**不是**撮合下单/即时匹配的交易平台，而更像一个**关系与账目的管理后台**：玩家和服务者之间已经建立了陪玩关系，平台负责记录余额、游玩记录、评价，并管理服务者与工作室的归属关系。

一句话概括三方关系：
- **工作室** 招募/审批 **服务者** 加入；
- **服务者** 为 **玩家** 提供陪玩服务，并管理玩家在自己这里的余额；
- **玩家** 查看自己的余额与游玩记录、给服务者/工作室评价。

---

## 2. 三种角色（设计 persona 用）

| 角色 | 是谁 | 核心诉求 | 高频操作 |
|------|------|----------|----------|
| **玩家 Player** | 购买陪玩服务的用户 | 看清「我还有多少余额、玩了什么、花了多少」 | 查余额、看游玩记录、写评价 |
| **服务者 Provider** | 提供陪玩的人 | 管理「我服务的每个玩家的账户」、决定是否挂靠工作室 | 给玩家加/扣余额、看收益汇总、申请加入工作室 |
| **工作室 Studio** | 管理多名服务者的组织 | 招募与审批服务者、统一运营 | 审批入驻申请、管理成员、查看整体经营 |

> 说明：
> - 服务者可以「独立」存在（不挂靠任何工作室）。
> - 当前没有独立的「管理员」角色，管理职责暂时由工作室角色兼任。
> - 一个账号注册时三选一确定角色。

## 3. 核心概念（界面要展示的对象，用大白话讲）

| 概念 | 含义 | 界面里怎么体现 |
|------|------|----------------|
| **余额 Balance** | 玩家在「某个服务者」处的账户余额。注意：余额是按「玩家×服务者」分别记的，不是一个全平台统一钱包。 | 一个玩家可能有多条余额（在不同服务者那里各一条） |
| **余额类型** | 三种：**金额**(money) / **时间**(time，按分钟) / **点数**(point) | 同一段关系里可能同时存在多种类型，需要分别展示 |
| **交易流水 Transaction** | 余额的每一次变动记录：充值/消费/退款/冻结/解冻 | 账单/流水列表，含变动前后金额、操作人、时间 |
| **游玩记录 PlayRecord** | 一次陪玩的记录：游戏名、模式、开始/结束时间、时长、消费金额、状态 | 时间线 / 列表，区分「进行中 / 已完成 / 已取消」 |
| **评价 Review** | 玩家对服务者或工作室的评价：1–5 星、文字、标签、可匿名 | 评分 + 标签 + 评论，支持匿名 |
| **入驻关系 Relation** | 服务者申请加入工作室的关系，状态：待审核/已批准/已拒绝 | 申请表单 + 审批列表 |

## 6. 关键用户流程（画原型时的主线）

**A. 玩家了解自己的账户**
注册→进入 Dashboard 看到余额概览→点进「我的余额」看到在各服务者处的余额→点某条余额看流水账单。

**B. 服务者给玩家记账**
登录→「玩家账户管理」→选中某玩家→发起一笔充值/扣费（选类型 money/time/point、填金额、备注）→流水里生成一条记录。

**C. 服务者加入工作室**
浏览「工作室列表」→进入某工作室详情→提交加入申请（填备注）→在「我的工作室归属」看到「待审核」→工作室审批后状态变「已批准/已拒绝」。

**D. 工作室审批入驻**
工作室在「入驻申请审批」看到待审批列表→查看申请者信息→批准或拒绝（可填备注）→申请者状态更新。

**E. 玩家评价**
游玩记录完成后→对服务者/工作室发起评价（1–5 星 + 标签 + 文字，可匿名）。

## 7. 给设计师的开放问题（需要产品/设计一起拍板）

1. **余额的心智模型**：现在余额是「玩家×服务者」分开记的（一个玩家在不同服务者那里各有独立余额）。设计上要不要把它表现成「一个钱包 vs 多个独立账户」？这直接影响玩家端首屏。
2. **三种余额类型（金额/时间/点数）如何并列展示**？是否所有服务者都同时用三种，还是大多只用其中一种？
3. **Dashboard 是否需要彻底按角色分化**？三种角色看到的首屏差异有多大。
4. **移动端优先还是桌面优先**？目前是桌面 Web（Ant Design 后台风格），但陪玩用户可能更偏移动端。
5. **整体气质**：现在是中性 SaaS 后台风。陪玩是偏娱乐/游戏社区的场景，视觉语言要不要更年轻化、更有「游戏感」？
6. **空状态/新手引导**：大量列表页（余额、记录、评价）新用户都是空的，空状态设计很重要。
7. **服务者「加/扣余额」是敏感的钱相关操作**，需要怎样的二次确认 / 防误操作设计？
